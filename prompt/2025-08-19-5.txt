> 怎麼Set up PostgreSQL database? 可以部署到zeabur自動設定?

⏺ 是的！Zeabur 可以自動設定 PostgreSQL 資料庫。以下是完整的設定步驟：

  2. 點選 "New Project"

  3. 新增 PostgreSQL 服務

  在 Zeabur 專案中：
  1. 點選 "Add Service"

  4. 設定環境變數

  DATABASE_URL=<Zeabur會自動提供的PostgreSQL連接字串>

  GOOGLE_CLIENT_SECRET=your_google_client_secret

  1. 在 Zeabur 找到 PostgreSQL 服務
  2. 複製連接字串
  3. 使用 PostgreSQL 客戶端執行：
  -- 貼上 database-schema.sql 的內容

> 為什麼有2個package.json檔?

⏺ 這是因為這個專案使用了 Monorepo 架構，將前端和後端分開管理：

  📁 專案結構說明

  code-playground/
  ├── package.json          # 根目錄 - 管理整個專案
  ├── client/
  │   └── package.json      # 前端依賴 (React, Vite, Tailwind)
  └── server/
      └── package.json      # 後端依賴 (Express, PostgreSQL, Passport)

  🎯 每個 package.json 的用途

  1. 根目錄的 package.json

  - 管理整個專案的腳本
  - 提供便捷的開發命令
  - 安裝 concurrently 同時啟動前後端

  2. client/package.json

  - React 前端依賴
  - Vite 建置工具
  - Monaco Editor、Tailwind CSS 等

  3. server/package.json

  - Node.js 後端依賴
  - Express、PostgreSQL、Passport 等

  🚀 使用方式

  # 安裝所有依賴
  npm run install:all

  # 同時啟動前後端
  npm run dev

  # 個別啟動
  npm run client:dev  # 只啟動前端
  npm run server:dev  # 只啟動後端

  💡 為什麼這樣設計？

  1. 分離關注點 - 前後端依賴完全分開
  2. 獨立部署 - 可以分別部署到不同服務
  3. 開發效率 - 團隊可以專注各自領域
  4. 版本控制 - 前後端可以有不同的版本週期

  這是現代全端專案的標準做法！

> 怎麼在本地端看此專案的網頁

⏺ 要在本地端看此專案的網頁，請按照以下步驟：

  🚀 本地開發設定步驟

  1. 安裝依賴

  # 在根目錄執行
  npm run install:all

  2. 設定資料庫

  # 安裝 PostgreSQL (如果還沒有)
  brew install postgresql
  brew services start postgresql

  # 建立資料庫
  createdb code_playground

  # 執行資料庫 schema
  psql -d code_playground -f database-schema.sql

  3. 設定環境變數

  # 複製環境變數範例
  cp server/.env.example server/.env

  編輯 server/.env 檔案：
  DATABASE_URL=postgresql://username:password@localhost:5432/code_playground
  GOOGLE_CLIENT_ID=your_google_client_id
  GOOGLE_CLIENT_SECRET=your_google_client_secret
  SESSION_SECRET=any_random_string_here
  CLIENT_URL=http://localhost:5173
  NODE_ENV=development

  4. 啟動開發伺服器

  # 同時啟動前後端
  npm run dev

  這會同時啟動：
  - 後端: http://localhost:5000 (API 服務)
  - 前端: http://localhost:5173 (React 網頁)

  -- Code Playground Database Schema

  -- Users table for Google OAuth authentication
  CREATE TABLE users (
      id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      google_id VARCHAR(255) UNIQUE NOT NULL,
      email VARCHAR(255) UNIQUE NOT NULL,
      name VARCHAR(255) NOT NULL,
      avatar_url TEXT,
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
      updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  );

  -- Playgrounds table to store user code snippets
  CREATE TABLE playgrounds (
      id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      user_id UUID REFERENCES users(id) ON DELETE CASCADE,
      title VARCHAR(255) NOT NULL DEFAULT 'Untitled',
      description TEXT,
      language VARCHAR(50) NOT NULL CHECK (language IN ('java', 'python')),
      code TEXT NOT NULL,
      is_public BOOLEAN DEFAULT FALSE,
      share_token VARCHAR(255) UNIQUE,
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
      updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  );

  -- Execution results table to store code execution history
  CREATE TABLE executions (
      id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      playground_id UUID REFERENCES playgrounds(id) ON DELETE CASCADE,
      code TEXT NOT NULL,
      output TEXT,
      error TEXT,
      execution_time_ms INTEGER,
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  );

  -- Create indexes for better query performance
  CREATE INDEX idx_playgrounds_user_id ON playgrounds(user_id);
  CREATE INDEX idx_playgrounds_share_token ON playgrounds(share_token);
  CREATE INDEX idx_playgrounds_is_public ON playgrounds(is_public);
  CREATE INDEX idx_executions_playground_id ON executions(playground_id);
  CREATE INDEX idx_users_google_id ON users(google_id);
  CREATE INDEX idx_users_email ON users(email);

  -- Add triggers to automatically update the updated_at timestamp
  CREATE OR REPLACE FUNCTION update_updated_at_column()
  RETURNS TRIGGER AS $$
  BEGIN
      NEW.updated_at = CURRENT_TIMESTAMP;
      RETURN NEW;
  END;
  $$ language 'plpgsql';

  CREATE TRIGGER update_users_updated_at BEFORE UPDATE ON users
      FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

  CREATE TRIGGER update_playgrounds_updated_at BEFORE UPDATE ON playgrounds
      FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

⏺ 是的！DATABASE_URL 和 POSTGRES_CONNECTION_STRING 是一樣的內容。

  🔗 Connection String 說明

  在 Zeabur 中：
  - POSTGRES_CONNECTION_STRING = Zeabur PostgreSQL 服務提供的連接字串
  - DATABASE_URL = 你的應用程式需要的環境變數名稱

  它們的值是相同的，格式通常像這樣：
  postgresql://username:password@host:port/database_name

  ✅ 設定步驟

  1. 複製 PostgreSQL Connection String

  從 PostgreSQL 服務頁面複製連接字串

  2. 在 code-playground 服務設定 DATABASE_URL

  Key: DATABASE_URL
  Value: <剛才複製的 PostgreSQL Connection String>

  🎯 為什麼要這樣設定？

  - PostgreSQL 服務：提供資料庫和連接資訊
  - code-playground 服務：需要透過 DATABASE_URL 環境變數來連接資料庫
  - 兩者必須使用相同的連接字串才能正確連接

  所以簡單來說：是的，DATABASE_URL 的值就是 PostgreSQL Connection String！

  設定完成後，你的應用程式就能成功連接到資料庫了。

  🏷️ Conventional Commits 類型

  - fix: - 修正 bug
  - feat: - 新功能
  - build: - 影響建置系統或外部依賴的變更 ✅
  - ci: - CI 配置文件和腳本的變更
  - docs: - 文檔變更
  - style: - 不影響程式碼意義的變更（空白、格式化等）
  - refactor: - 既非修正 bug 也非新增功能的程式碼變更
  - perf: - 改善效能的程式碼變更
  - test: - 新增測試或修正現有測試
  - chore: - 對建置流程或輔助工具的變更
